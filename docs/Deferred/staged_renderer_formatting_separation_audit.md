# Staged Renderer Formatting Separation Audit

## Scope

This note answers two questions about the current staged renderer paths:

1. How well is output formatting separated from rendering logic?
2. Are there `.css` files that map visual appearance such as fill colors, line styles, and font attributes?

The audit is based on the current staged renderer pipeline and shared renderer infrastructure in `src/renderer/staged/`.

## Short Answer

- Separation is moderate, not strict.
- There are no standalone `.css` files in the repository for staged renderer appearance mapping.
- Visual appearance is mostly defined through:
  - shared theme tokens in `src/renderer/staged/theme.ts`
  - SVG class emission in `src/renderer/staged/svgBackend.ts`
  - an embedded `<style>` block generated by `src/renderer/staged/svgBackend.ts`
- The scene-building layers mostly emit semantic roles and class names rather than raw paint attributes.
- The SVG backend still owns the class-to-style mapping, including some view-specific overrides, so styling is centralized but not externalized.

## Evidence Summary

### Pipeline Boundary

The staged pipeline is explicit:

- `RendererScene`
- `MeasuredScene`
- `PositionedScene`
- SVG backend
- PNG rasterized from SVG

This is implemented in `src/renderer/staged/pipeline.ts`.

That is a strong architectural separation between semantic scene construction, measurement/layout, and final backend emission.

### No Standalone CSS Files

A repository search for `*.css` returned no matches.

For staged SVG output, CSS is generated inside the backend and embedded into the SVG `<defs>` section rather than loaded from a separate stylesheet file.

## What Is Cleanly Separated

### 1. Semantic Scene Construction Versus Final Paint Attributes

The scene contracts carry semantic fields such as:

- `primitive`
- `role`
- `classes`
- `themeId`

They do not carry raw SVG paint attributes such as `fill`, `stroke`, `font-size`, or `stroke-dasharray`.

Relevant definitions:

- `src/renderer/staged/contracts.ts`

Example:

- `SceneContainer`, `SceneNode`, and `SceneEdge` all carry `classes: string[]`
- `RendererScene` carries `themeId`

This is a good separation boundary.

### 2. Scene Builders Mostly Emit Semantic Classes

View builders such as:

- `src/renderer/staged/serviceBlueprint.ts`
- `src/renderer/staged/uiContracts.ts`
- `src/renderer/staged/iaPlaceMap.ts`

primarily emit:

- roles
- semantic classes
- width policy
- overflow policy
- routing intent
- content blocks

For example:

- `serviceBlueprint.ts` assigns classes like `service_blueprint_node`, `shape-*`, `type-*`, and style-token-derived classes such as `chrome-dashed`
- `uiContracts.ts` emits semantic classes and style-token-derived classes

The builders generally do not hardcode paint values.

### 3. Theme Tokens Are Centralized

`src/renderer/staged/theme.ts` centralizes several formatting and appearance concerns:

- font family and font asset paths
- text styles
- width bands
- edge label width limits
- primitive padding and spacing
- primitive minimum heights
- corner radii
- stroke widths
- arrow size
- color palette

This is the main source of reusable renderer formatting policy.

## What Is Only Partially Separated

### 1. The SVG Backend Owns the Style Mapping

`src/renderer/staged/svgBackend.ts` generates classed SVG markup and also generates the CSS rules that define what those classes mean.

This happens in `buildStyleLines(...)` and `buildDefs(...)`.

That means:

- semantic classes are separated from scene-building logic
- but the mapping from those classes to actual appearance still lives inside rendering code

So the separation is better described as:

- scene builders are separated from paint values
- backend rendering and style authoring are still combined

### 2. View-Specific Presentation Rules Still Live in the Backend

`svgBackend.ts` includes view-specific selectors such as:

- `.view-service_blueprint ...`
- `.scene-container.role-view_state ...`

These are presentation behaviors, not generic SVG serialization.

Examples include:

- hiding `service_blueprint_cell` chrome
- making `service_blueprint_band_column` chrome transparent
- styling service blueprint separator lines
- changing `view_state` container fill behavior

This is the clearest place where output formatting is still mixed into backend logic.

### 3. Typography and Layout Are Intentionally Coupled Through the Theme

The current theme is not only a paint theme.

Measurement and layout also depend on it via:

- `textStyles`
- primitive padding
- primitive minimum heights
- width bands
- edge label width limits

`src/renderer/staged/microLayout.ts` resolves text styles from the theme during measurement, and uses the measurement font asset to wrap and size text.

This is appropriate for deterministic rendering, but it means "formatting" is broader than paint and is not cleanly separable from layout behavior.

## Theme-Driven Concerns Versus Backend-Hardcoded Concerns

### Centralized In `theme.ts`

These are centralized in the shared renderer theme:

- font family
- font asset locations for measurement, SVG, and PNG
- text style roles such as `title`, `subtitle`, `badge`, `metadata`, `label`, and `edge_label`
- width band values
- edge label max width
- primitive padding and spacing defaults
- primitive minimum heights
- port inset defaults
- badge padding
- corner radii
- canvas background
- stroke widths
- edge stroke width
- port radius
- arrow size
- palette colors for containers, nodes, badges, ports, text, edges, and edge labels

These tokens are reused by measurement and backend emission.

### Hardcoded In `svgBackend.ts`

These remain hardcoded in backend code rather than being data-driven theme or stylesheet definitions:

- the CSS selector structure itself
- the mapping from semantic classes to paint rules
- dash-array recipes for dashed and dotted chrome and edges
- the rule that bold edges increase stroke width by `+1`
- the selector rules for metadata and secondary text coloring
- the selector rules for `role-view_state`
- the selector rules for `view-service_blueprint`
- the fact that service blueprint cell chrome is hidden
- the fact that service blueprint separator lines use `stroke-dasharray: 6 4`
- the marker definitions and their geometry

These choices are centralized, but they are backend-owned implementation details rather than theme data.

### Hardcoded Outside Theme And Outside SVG Backend

Some formatting-adjacent behavior lives in scene builders or decoration code:

- per-view root padding and gaps
- service blueprint left gutter and column gap
- service blueprint cell padding and gap
- UI contracts scope padding and root gap
- service blueprint lane title positions and separator label positions

Examples:

- `src/renderer/staged/serviceBlueprint.ts`
- `src/renderer/staged/uiContracts.ts`
- `src/renderer/staged/serviceBlueprintDecorations.ts`

These are mostly layout and presentation-structure constants rather than paint rules.

## Current Practical Model

The current staged renderer is effectively using a three-layer presentation model:

1. Semantic scene layer
   - builders emit roles, primitives, content blocks, routing intent, and class tokens
2. Shared theme layer
   - shared tokens define typography, spacing, geometry defaults, and palette values
3. SVG backend style layer
   - backend code maps semantic classes and view classes to concrete SVG/CSS output

That is a reasonable internal architecture, but it is not the same as having:

- a separate stylesheet file
- a declarative theme-to-selector mapping layer
- a backend that only serializes geometry

## Answer To The CSS Question

No. There are currently no `.css` files that serve as the staged renderer's appearance map.

Instead, appearance is produced by generated CSS embedded directly into each SVG:

- `@font-face` is generated by `src/renderer/svgArtifacts.ts`
- scene styling rules are generated by `src/renderer/staged/svgBackend.ts`

## Assessment

### Strengths

- Good semantic separation between scene construction and raw paint attributes
- Shared theme tokens are centralized
- Deterministic font handling is built into both measurement and final rendering
- SVG remains the primary vector artifact and PNG is derived from it

### Weak Spots

- The backend still mixes SVG serialization with stylesheet authoring
- Some view-specific appearance rules are hardcoded in the shared backend
- Not all presentation tokens are theme-driven
- Layout formatting and visual formatting are intentionally intertwined through measurement

## Bottom Line

The staged renderer does separate semantics from final paint values better than the legacy paths, but output formatting is not fully isolated from rendering logic.

The codebase currently has:

- centralized theme tokens
- centralized generated SVG styles
- no standalone CSS files

The main remaining separation gap is that `src/renderer/staged/svgBackend.ts` still serves both as:

- the SVG serializer
- the authoritative stylesheet generator for staged output
